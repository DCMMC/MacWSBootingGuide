Goal context: get a WORKING native-iOS AGX command-buffer trace to byte-diff against the failing macOS-AGX (agxprobe) path that submits but never executes (status 5 / 0x103). See [[agx-direct-path-kernel-abi-deadend]].

**BREAKTHROUGH (2026-06-14): a TrollStore-installed app with `platform-application` gets REAL M1 GPU access.**
`misc/iosblit.m` built as a UIKit `.app` (com.dcmmc.iosblit), installed via TrollStore's `trollstorehelper install <ipa>`, launched via `uiopen --bundleid`, logged: `device=Apple M1 GPU`, blit `status=4` (Completed), `readback=0xab`. The GPU actually executed the command buffer. This is the working iOS reference path on the real AGX.

**Why TrollStore is required (ad-hoc fails):**
- Bare SSH-exec'd binary: `MTLCreateSystemDefaultDevice()` = nil (no GPU sandbox extension).
- uicache-registered ad-hoc app: RunningBoard grants the GPU assertion ("Set Darwin GPU to allow"), but the sandbox MACF still denies it: `System Policy: iosblit deny(1) iokit-open-user-client AGXDeviceUserClient`, IOServiceOpen returns 0xE00002E2 (kIOReturnNotPermitted). `platform-application` would bypass this but amfid does NOT honor it on an ad-hoc signature.
- TrollStore: CoreTrust bypass re-signs with an App Store code directory (TeamID) so amfid HONORS `platform-application` → sandbox treats it as a platform binary → AGXDeviceUserClient allowed. Entitlements that survived install (dumped by trollstorehelper): application-identifier, get-task-allow, platform-application, container-required.

**Build/deploy recipe:** `xcrun -sdk iphoneos clang -arch arm64e -fmodules misc/iosblit.m -framework Foundation -framework Metal -framework IOKit -framework UIKit -o Payload/iosblit.app/iosblit`; `ldid -S<ent>` (ent = application-identifier + get-task-allow + platform-application); zip Payload → ipa; push to device; `/var/containers/Bundle/Application/<TS-UUID>/TrollStore.app/trollstorehelper install /path.ipa`. App must be UIKit (creates a foreground window) so it holds the GPU sandbox extension; do the Metal work in didFinishLaunching. Log via os_log (funopen-redirect stderr → os_log subsystem com.dcmmc.iosblit), read with `oslog | grep`.

**CAPTURE PROBLEM (unsolved):** the app's built-in `__DATA,__interpose` tracer catches NOTHING — Metal.framework is in the dyld shared cache, and shared-cache-internal binds (Metal→IOKit) bypass main-executable dyld interposing on iOS. (This is exactly why libmachook works in the chroot: there AGXMetal loads from a FILE, so its binds are interposable.)

**DANGER — do NOT repeat:** an ellekit tweak (`misc/iosblit_tracer.m`) using `MSHookFunction` to inline-hook hot IOKit funcs (IOConnectCallStructMethod/Method/Async) at app launch DEADLOCKED the app uninterruptibly: the render-server thread calls those constantly during UIKit launch, and patching a shared-cache function mid-execution races → process stuck in kernel (survives kill -9). This WEDGED the AGX GPU: after the stuck process, even clean tweak-off launches hang pre-main (UIKit render-server GPU init blocks). Wedge persisted across process reaping → needed device recovery. Repeated GPU launches risk full device crash (cf. [[agx-direct-path-kernel-abi-deadend]]).

**DECISIVE (2026-06-14) — the +0x00 tag is a PAC-SIGNED POINTER (runtime signature, varies per-run):** re-captured the iOS cmdbuf +0x00 across 2 runs: RUN1 cmdbuf0=0x001e80`0207562b98 (tag 0x1e80), RUN2 cmdbuf0=0x007c00`0207562b98 (tag 0x7c00). The LOW 40 bits are CONSTANT (0x0207562b98 = a shared-cache structure ptr, fixed this boot) but the HIGH 16 bits (bytes +0x05/+0x06) VARY per-cmdbuf AND per-run = the textbook signature of an arm64e PAC-signed pointer (PAC = ~16 signature bits in the high bits, varying with the discriminator/context). The macOS-chroot driver leaves +0x00 RAW/unsigned (0x00000002`02717770, no high tag). So the 0x103 is a **PAC AUTH FAILURE**: the iOS kernel/firmware autia's the cmdbuf-descriptor +0x00 pointer; iOS (signed) passes, raw-macOS faults → reject. **MAJOR REFRAME (2026-06-14) — the PAC tag is a generic ObjC-Block isa signature, NOT AGX/firmware integrity:** decompiled `-[IOGPUMetalCommandBuffer fillCommandBufferArgs:commandQueue:]` (iOS IOGPU @0x1eec66fb0). It builds a STACK BLOCK (used as kernel-side callback), heap-copies it via `bl 0x1f3027e80` (= objc_msgSend$<sel>, probably `[block copy]`), and stores the heap-block ptr at args+0x10 / args+0x18. The PAC instructions are STANDARD arm64e Block-runtime construction:
- `pacda x16, x17` with disc = `(stack_addr_of_isa_slot) | (0x6ae1ULL << 48)` → signs **Block.isa** at cb+0x00
- `pacia x16, x8` with disc = stack_addr_of_invoke_slot → signs **Block.invoke** at cb+0x10
The macOS IOGPU build has the **IDENTICAL pacda/pacia instructions** (same disc constant `0x6ae1`, confirmed via objdump of /tmp/macos-iogpu/IOGPU at `19d143d54: movk x17,#0x6ae1,lsl #48; pacda x16,x17`). So the cmdbuf descriptor passed to the kernel is just an ObjC Block — its +0x00 isa and +0x10 invoke are signed by the standard Block runtime in BOTH builds.

**SO WHY does macOS-chroot dump show RAW (unsigned) values for isa+invoke?** Captured macOS cmdbuf0: isa=0x0000000202717770 (high 16 = 0x0000), invoke=0x00000001c445be14 (high 16 = 0x0000). On iOS native: isa=0x001e80020756_2b98 (high tag varies), invoke=0x757a0002190ff0fc (also tagged). Same source code emits `pacda` in both — but the chroot result has NO PAC tag in the high bits. Possibilities: (a) the chroot's libsystem_blocks `_Block_copy` is older/different and skips re-signing on heap-copy; (b) the chroot is on the arm64e-preview ABI where PAC writes to a different bit range; (c) the spoofed-macOS process keys (DA/IA) are unset/zero in the chroot launchdchrootexec context so `pacda` is a no-op; (d) something strips tags via xpacd between sign and submit. **(c) is the leading suspect** — launchdchrootexec may not propagate the arm64e per-task keys properly, so `pacda` returns the input ptr unchanged → unsigned isa → kernel `autda` fails → 0x103.

**PAC KEY FIX FOUND (2026-06-14): cpusubtype USR01→USR00 (one byte) makes iOS kernel set up real PAC keys for chroot processes.** Empirical chain (pactest.m): same launchdchrootexec + chroot + setuid + execve path runs (a) iOS-built arm64e binary (caps=0x80, USR00) → PAC works, AND (b) macOS-built arm64e binary (caps=0x81, USR01) → DA/IA/DB zero, only IB works. The launchdchrootexec / chroot / posix_spawn path is innocent; the kernel deliberately disables 3 of 4 PAC keys for "USR01" (macOS arm64e ABI variant) binaries. Patching the cpusubtype caps byte at Mach-O header offset 0x0b from 0x81 to 0x80 makes the iOS kernel treat the binary as USR00 (iOS arm64e ABI) and provision real DA/IA/DB/IB per-task keys. Confirmed: patched pactest_usr00 produces pacda delta=0x21800000000000 (non-zero) inside chroot — all 4 keys functional. macOS dyld in chroot loads USR00 binaries without complaint. Patch tool: misc/set_ios_version.py (LC_BUILD_VERSION macOS→iOS) + a 1-byte caps poke (`d[11]=0x80`). NOTE on ripple effects: with PAC keys active, dlsym now returns PAC-signed function pointers — code that treated dlsym results as raw integers (e.g. agxprobe's patch_sandbox_gpu_trust doing `(uintptr_t)ft + 0x190`) must call `ptrauth_strip(ft, ptrauth_key_asia)` first; otherwise EXC_BAD_ACCESS on deref.

**0x103 = KERNEL VALIDATION FAILURE (2026-06-14, DECISIVE FIND).** Extracted com.apple.AGXG13G + com.apple.AGXFirmwareKextG13GRTBuddy + com.apple.driver.RTBuddy from kernelcache.decompressed via `ipsw kernel extract <kc> <kext> -o <dir>`. ALL FOUR `mov w?, #0x103` immediates in AGXG13G live inside ONE function: **`AGXCommandQueue::processSegmentKernelCommand`** (@0xfffffe00086e2274). Pattern at each site (e.g. +0x1884):
```
ldr w9, [sp, #...d8]   ; cmd->type (offset +0x08 in 16B header)
cmp w9, #0x30          ; must equal 0x30
b.ne <type-err-path>   ; NOT 0x103 — different code path
ldr w10, [sp, #...d4]  ; cmd->size (+0x04)
mov w9, #0x103         ; ★ 0x103 staged
cmp w10, #0x1a8        ; ★ size MUST equal 0x1a8 (424 bytes) — ELSE 0x103
b.ne err_path
ldr w11, [sp, #...d0]  ; cmd->end_offset (+0x00)
add x10, x10, #0x30    ; size + 0x30 (header)
cmp x10, x11           ; end_offset MUST equal size+0x30
b.ne err_path
```
The 16-byte header is loaded from `x9+0x28` (`ldur q0, [x9, #0x28]; str q0, [sp, #0x1d0]`). x9 is a per-segment context object whose +0x28 is the per-command-block 16-byte header. So **0x103 = "kernel command size != 0x1a8 OR end_offset != size+0x30"** in `AGXCommandQueue::processSegmentKernelCommand`. The 4th site is a slightly different range check (sub-field at `[x28, #0x30]` must be <= 2 AND match certain layout). This is KERNEL-SIDE validation, NOT firmware. The kernel processes the cmdbuf args BEFORE forwarding to firmware; if validation fails, the kernel sets 0x103 itself and never submits to firmware.

**CORRECTS earlier "firmware-relayed" assumption:** earlier I said "the kernel just relays 0x103 from firmware via shared memory" because the IOGPUFamily kext (the GENERIC one I had) has zero `mov w?, #0x103`. That was because the AGX-family-specific kext (AGXG13G) wasn't extracted yet — and 0x103 is set in AGXG13G's processSegmentKernelCommand. So it IS kernel-side, IS based on cmdbuf content, and **IS bridgeable by a userland shim that gets the cmdbuf content right.**
