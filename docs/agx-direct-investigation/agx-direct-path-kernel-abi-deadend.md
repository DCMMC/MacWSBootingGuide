**KEY TECHNIQUE that cracked it:** the iOS kernel's IOGPUFamily os_log is NOT redacted for these messages — `oslog` captured `kernel_task: IOGPUResource::newResourceWithOptions(...): invalid arguments` + `Terminal: Failed to allocate heap of size 4096, reason: failed to create IOGPUResourceRef`. That redirected from "memory/budget" (all wrong) to ARGUMENT VALIDATION.

**ROOT CAUSE (found):** `newResourceWithOptions` top check (kext 0xf4eb8): `cbz x4 -> "invalid arguments" -> NULL`. x4 = 5th arg = **IOByteCount = IOGPUNewResourceArgs+0x40**. The macOS AGX driver leaves args[0x40]=0 (stores size elsewhere); iOS requires it non-zero. (Also 0xf4ec4: `cbnz (x3>>32)` where x3=args[0x18] must fit 32 bits — ours is 0, ok.)

**FIX PROVEN:** in IOConnectCallMethod_new, for sel 0x9 (new_resource) when args[0x40]==0, set it to 0x1000 → that new_resource call returns **0x0 (success)** instead of 0xe00002be. Confirmed live.

**REMAINING (next):** AGX issues MULTIPLE new_resource calls. Two kinds seen:
- args[0]=0x00, args[0x40]=0 → FIXED by the above (→0x0).
- **args[0]=0x80** (bit7 set) → routed to `IOGPUResource::newResourceWithParentResourceID` (kext 0xef533c, new_resource 0xf03d50 `cmp #0x80`→0xf03d7c→0xf03e18). These are SUB-RESOURCES referencing a parent heap at GPU VA offsets up to ~0x10ced0000 (~4.5GB) — fields args[0x38]/[0x40] hold VAs, args[0x30]=parent offset, args[0x48]=id. They return 0xe00002c2 (BadArgument) because my args[0x40]=0x1000 hack made the PARENT heap only 4KB, so the sub-resource's multi-GB offset is out of bounds.

**SO THE FIX MUST BE:** set args[0x40] (heap byte-count, for args[0]=0 heaps) to the REAL intended heap size (large/virtual, ≥ the max sub-resource offset, e.g. the 0xc38000000-class virtual size the macOS driver computes), NOT a hardcoded 0x1000. Find the offset where the macOS AGX driver stores the heap size (it left iOS's 0x40 zero) and copy it to 0x40 — OR set a large value. Then re-test: both the heap (args[0]=0) AND the sub-resource (args[0]=0x80) should succeed. Then expect more types / command-queue (sel 7, 0x408 struct).

**UPDATE — byte-count must be 32-bit; sub-resource is a parent-ID lookup:**
- args[0x40] = 0xc38000000 (full u64 of +0x58) → heap returns BadArgument: IOByteCount is 32-bit, >32 bits rejected. The CURRENT mac_hooks.m fix copies only the LOW 32 bits of +0x58 (0x38000000=896MB) → heap succeeds. (0x1000 also worked for the heap; sub-resource size is unrelated to heap size.)
- args[0]=0x80 sub-resource → `newResourceWithParentResourceID` (0xef533c) → `bl IOGPUDevice::getResource(parentID)` at 0xf5488; `cbz x0 → fail` (parent not found) → BadArgument. Then checks parent->[0x14] flags (`bics xzr,x23,parent[0x14]; b.ne fail`) and parent->[0x10] (`(p[0x10]|0x80)!=0x80 → fail`). So the sub-resource references a PARENT resource by ID; the lookup fails because the AGX driver's expected parent ID ≠ the ID the iOS kernel assigned to the heap (returned in the heap new_resource's 80-byte outStruct).
- **NEXT:** capture (a) the heap new_resource's OUTSTRUCT (80B — the kernel-assigned resource ID/handle) and (b) the sub-resource's parent-ID arg field, and reconcile them.
